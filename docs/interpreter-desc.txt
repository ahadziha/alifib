❯ explain to me how the interpreter state is structured                         

⏺ The interpreter state has two layers: a context that travels with each step,
  and a global store that holds everything persistent.

  ---
  Context

  pub struct Context {
      pub current_module: String,
      pub state: Arc<GlobalStore>,
  }

  The Context is passed by value (cloned) through every interpreter step — it's
  the interpreter's "current position". It knows which module is being processed
   and holds a shared pointer to all persistent state. Mutations go through
  Arc::make_mut(&mut context.state), which implements copy-on-write: if nothing
  else holds a reference to the store, it mutates in place; otherwise it clones
  first.

  ---
  GlobalStore

  pub struct GlobalStore {
      pub cells:        HashMap<GlobalId, CellEntry>,
      pub cells_by_dim: HashMap<usize, Vec<GlobalId>>,
      pub types:        HashMap<GlobalId, TypeEntry>,
      pub modules:      HashMap<ModuleId, Arc<Complex>>,
  }

  There are three distinct namespaces:

  cells — higher-dimensional cells (1-cells and above) declared inside type
  complexes. Each entry just stores the cell's CellData (its boundary
  information). Indexed by GlobalId.

  types — 0-dimensional type generators, declared in @Type blocks. Each
  TypeEntry stores its CellData (always Zero since they're 0-dimensional) and
  its type complex — the environment of generators, diagrams, and maps declared
  inside its { ... } body.

  modules — each module (.ali file) gets its own Complex, also stored here. Both
   types and modules store their associated environment as a Complex, but via
  different tables because they have different roles.

  ---
  Complex

  pub struct Complex {
      generators:  Generators,               // name ↔ tag ↔ dim ↔ classifier
      diagrams:    HashMap<LocalId, Diagram>,
      maps:        HashMap<LocalId, MapEntry>,
      local_cells: LocalCells,
      used_names:  HashSet<LocalId>,
  }

  A Complex is the local environment for either a type or a module. It stores:

  - Generators with three indices: by name (by_name → GeneratorEntry{tag, dim}),
   by tag (by_tag → name), and by dimension (by_dim). Generators also each have
  a classifier diagram — a single-cell diagram whose sole label is that
  generator's Tag.
  - Diagrams — named diagram bindings (let f = ...).
  - Maps — named partial map bindings (def f : A = ...), each with a MapEntry { 
  map: PMap, domain: MapDomain }.
  - Local cells — cells with Tag::Local(name) instead of a GlobalId, used inside
   @Local blocks where the cell identity doesn't persist globally.

  ---
  CellData and Tags

  Every cell (whether in cells, types, or local) has CellData:

  pub enum CellData {
      Zero,
      Boundary { boundary_in: Arc<Diagram>, boundary_out: Arc<Diagram> },
  }

  Zero for 0-cells (objects), Boundary for higher cells (the source and target
  are diagrams one dimension lower).

  Every generator has a Tag that identifies it:

  pub enum Tag { Global(GlobalId), Local(LocalId) }

  Global tags refer to entries in the GlobalStore and persist across module
  boundaries. Local tags (strings) are ephemeral — they only exist within the
  scope of an @Local block and are never stored in the global state.

  ---
  How It Fits Together

  When the interpreter processes a type A:
  1. A fresh GlobalId is minted and state.types[id] is created with
  CellData::Zero and an initially-empty Complex.
  2. A classifier diagram (a 0-cell labeled Tag::Global(id)) is stored in the
  current module's complex under the name "A".
  3. As the body { ... } is processed, higher-dimensional generators and diagram
   bindings accumulate in state.types[id].complex.

  When it processes a cell f : A → B inside a type complex:
  1. A fresh GlobalId is minted for f.
  2. state.cells[f_id] is created with CellData::Boundary { boundary_in:
